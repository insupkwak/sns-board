# SNS 게시판 대규모 유저용 고도화 계획

> 기존 SNS 게시판을 1,000명 이상 사용자가 안정적으로 사용할 수 있는 서비스형 커뮤니티 플랫폼으로 확장합니다.
> 단순히 글을 쓰는 게시판이 아니라, 여러 게시판이 서로 노출되고, 사용자가 관심 게시판을 찾고, 게시판 운영자가 자신의 게시판을 홍보할 수 있는 구조를 추가합니다.

---

## 1. 고도화 목표

## 1.1 현재 상태

현재 SNS 게시판은 다음 기능을 갖춘 MVP 상태입니다.

| 구분   | 현재 기능                     |
| ---- | ------------------------- |
| 사용자  | 회원가입, 로그인, 로그아웃           |
| 게시글  | 본문 중심 글쓰기, 수정, 삭제         |
| 카테고리 | 자유, 질문, 정보, 일상, 유머        |
| 소통   | 좋아요, 댓글                   |
| 목록   | 최신순, 무한 스크롤               |
| 디자인  | 모바일 최적화, 심플한 SNS 스타일      |
| 배포   | Vercel 미리보기, Oracle 운영 예정 |
| DB   | SQLite 기반                 |

---

## 1.2 고도화 후 목표

고도화 후에는 다음 형태의 서비스로 확장합니다.

| 구분    | 목표                                        |
| ----- | ----------------------------------------- |
| 안정성   | 1,000명 이상 사용자 접속에도 서비스 유지                 |
| 확장성   | 게시글, 댓글, 좋아요 증가에도 속도 유지                   |
| 커뮤니티화 | 여러 게시판을 만들고 운영할 수 있는 구조                   |
| 유입 구조 | 게시판 홍보, 추천, 검색, 랭킹 기능 제공                  |
| 탐색성   | 사용자가 관심 있는 게시판을 쉽게 찾을 수 있음                |
| 운영성   | 관리자, 신고, 차단, 백업, 모니터링 기능 추가               |
| 배포 구조 | Oracle 서버 운영 기준으로 안정화                     |
| DB 구조 | SQLite에서 PostgreSQL 또는 MySQL로 전환 가능하도록 설계 |

## 2. 핵심 방향

이번 고도화는 단순 기능 추가가 아니라, 서비스 구조 자체를 다음 단계로 올리는 작업입니다.

```text
기존 구조
사용자 → 글 작성 → 카테고리별 피드

고도화 구조
사용자 → 게시판 탐색 → 게시판 가입 또는 팔로우 → 글 작성 → 홍보 → 추천 노출 → 신규 유저 유입
```

## 4. 대규모 유저 기준 정의

“1,000명 사용”은 두 가지로 나누어 생각해야 합니다.

| 기준            | 설명                | 난이도 |
| ------------- | ----------------- | --- |
| 가입자 1,000명    | 전체 회원 수가 1,000명   | 낮음  |
| 일간 사용자 1,000명 | 하루 방문자가 1,000명    | 중간  |
| 동시 접속자 1,000명 | 같은 시간에 1,000명이 접속 | 높음  |

이 프로젝트의 1차 목표는 다음 기준으로 설정합니다.

| 항목      |                 목표 |
| ------- | -----------------: |
| 전체 회원 수 |          1,000명 이상 |
| 일간 방문자  |          1,000명 수준 |
| 동시 접속자  |      50명에서 200명 수준 |
| 게시글 수   |        10만 개 이상 대비 |
| 댓글 수    |        50만 개 이상 대비 |
| 좋아요 수   |       100만 개 이상 대비 |
| 응답 속도   |       주요 API 1초 이내 |
| 장애 대응   | DB 백업, 로그, 모니터링 가능 |

동시 접속자 1,000명을 목표로 할 경우에는 단일 Oracle 무료 서버와 SQLite만으로는 부족할 수 있으므로, DB 분리, 캐시 서버, 로드밸런서 구조까지 고려해야 합니다.

## 5. 기술 구조 고도화

## 5.1 현재 구조

```text
Browser
  ↓
Flask app.py
  ↓
SQLite board.db
```

현재 구조는 단순하고 빠르게 만들 수 있지만, 사용자가 증가하면 다음 문제가 생길 수 있습니다.

| 문제            | 설명                            |
| ------------- | ----------------------------- |
| SQLite 쓰기 잠금  | 동시에 글, 댓글, 좋아요가 많이 발생하면 병목 가능 |
| 단일 app.py 비대화 | 기능 증가 시 유지보수 어려움              |
| 세션 저장 한계      | 여러 프로세스 운영 시 세션 관리 복잡         |
| 파일 업로드 부담     | 서버 디스크 관리 필요                  |
| 검색 성능 저하      | 게시글이 많아질수록 LIKE 검색 느려짐        |
| 관리자 기능 부족     | 신고, 차단, 삭제 이력 관리 어려움          |

---

## 5.2 권장 운영 구조 1단계

1,000명 규모 초기 운영은 다음 구조를 권장합니다.

```text
User Browser
  ↓
Nginx
  ↓
Gunicorn
  ↓
Flask App
  ↓
PostgreSQL 또는 MySQL
```

| 구성                  | 역할                       |
| ------------------- | ------------------------ |
| Nginx               | 정적 파일 제공, 리버스 프록시, HTTPS |
| Gunicorn            | Flask 앱 운영용 WSGI 서버      |
| Flask               | API 서버                   |
| PostgreSQL 또는 MySQL | 영속 데이터 저장                |
| 서버 디스크              | 첨부파일 저장                  |
| Cron                | 백업, 정리 작업                |
| 로그 파일               | 접속, 에러, 운영 로그            |

---

## 5.3 권장 운영 구조 2단계

트래픽이 더 늘어나면 캐시와 비동기 작업을 추가합니다.

```text
User Browser
  ↓
Nginx
  ↓
Gunicorn
  ↓
Flask App
  ├─ PostgreSQL
  ├─ Redis
  ├─ File Storage
  └─ Background Worker
```

| 구성                | 역할                         |
| ----------------- | -------------------------- |
| Redis             | 세션, 캐시, 인기글 계산, Rate Limit |
| Background Worker | 알림, 랭킹 계산, 이미지 처리          |
| File Storage      | 첨부파일 분리 저장                 |
| Scheduler         | 매시간 인기 게시판 계산              |
| Monitoring        | 서버 상태 확인                   |

## 6. DB 고도화 방향

## 6.1 SQLite 유지 가능 구간

SQLite는 다음 정도까지는 사용할 수 있습니다.

| 상황          | 가능 여부    |
| ----------- | -------- |
| 개인 테스트      | 가능       |

| 동시 쓰기 적음    | 가능       |
| Vercel 미리보기 | 임시 사용 가능 |

---

## 6.2 SQLite에서 전환이 필요한 시점

다음 상황이 오면 PostgreSQL 또는 MySQL 전환을 권장합니다.

| 상황                  | 전환 필요성 |
| ------------------- | ------ |
| 글, 댓글 작성이 동시에 자주 발생 | 높음     |
| 좋아요 클릭이 많음          | 높음     |
| 검색 기능이 자주 사용됨       | 높음     |
| 게시글이 10만 개 이상       | 높음     |
| 운영자가 여러 명           | 중간     |
| 알림, 팔로우, 추천 기능 추가   | 높음     |
| 실제 외부 유저 유입 시작      | 높음     |

---

## 6.3 권장 DB

| DB         |   추천도 | 설명                   |
| ---------- | ----: | -------------------- |
| PostgreSQL | 매우 높음 | 검색, 인덱스, 확장성, 안정성 우수 |
| MySQL      |    높음 | 운영 경험 많고 안정적         |
| SQLite     |    낮음 | MVP와 테스트용에 적합        |

운영 기준으로는 PostgreSQL을 권장합니다.

## 8.2 DB 인덱스 권장

게시글이 많아질수록 인덱스가 중요합니다.

```sql
CREATE INDEX IF NOT EXISTS idx_posts_board_created

ON posts(user_id, created_at DESC);

ON posts(category, created_at DESC);

ON comments(post_id, created_at ASC);

ON likes(post_id);

ON likes(user_id);

ON boards(slug);

ON boards(created_at DESC);

CREATE INDEX IF NOT EXISTS idx_board_follows_user
ON board_follows(user_id);

```

## 8.3 Cursor 기반 무한 스크롤

현재 offset 방식은 데이터가 많아질수록 느려질 수 있습니다.

### 기존 방식

```http
GET /api/posts?offset=1000&limit=20
```

### 개선 방식

```http
GET /api/posts?before_id=98231&limit=20
```

또는:

```http
GET /api/posts?cursor=2026-05-30T10:30:00&id=98231&limit=20
```

### 장점

| 항목     | 설명                  |
| ------ | ------------------- |
| 속도     | 뒤 페이지로 가도 빠름        |
| 안정성    | 새 글이 추가되어도 중복 로딩 감소 |
| 확장성    | 게시글 수가 많아져도 유리      |
| 모바일 UX | 무한 스크롤과 궁합 좋음       |

## 8.4 좋아요 수, 댓글 수 집계 방식 개선

현재는 매번 좋아요와 댓글 개수를 계산할 수 있습니다.
사용자가 늘어나면 집계 컬럼을 두는 것이 좋습니다.

### posts 테이블에 추가 권장

```sql
ALTER TABLE posts ADD COLUMN like_count INTEGER DEFAULT 0;
ALTER TABLE posts ADD COLUMN comment_count INTEGER DEFAULT 0;
ALTER TABLE posts ADD COLUMN view_count INTEGER DEFAULT 0;
```

### 처리 방식

| 행동     | 처리                                      |
| ------ | --------------------------------------- |
| 좋아요 추가 | likes insert, posts.like_count +1       |
| 좋아요 취소 | likes delete, posts.like_count -1       |
| 댓글 작성  | comments insert, posts.comment_count +1 |
| 댓글 삭제  | comments delete, posts.comment_count -1 |
| 글 조회   | posts.view_count +1 또는 비동기 집계           |

## 9. 게시판 시스템 확장

## 9.1 게시판 개념 추가

기존에는 하나의 SNS 피드만 있었습니다.
고도화 후에는 사용자가 여러 게시판을 만들고 운영할 수 있습니다.

```text
게시판
├─ 게시판 이름
├─ 게시판 설명
├─ 게시판 대표 이미지
├─ 태그
├─ 공개 여부
├─ 운영자
├─ 멤버 수
├─ 게시글 수
├─ 팔로워 수
└─ 홍보 상태
```

---

## 9.2 게시판 유형

| 유형      | 설명              |
| ------- | --------------- |
| 공개 게시판  | 누구나 보고 글 작성 가능  |
| 가입형 게시판 | 가입 후 글 작성 가능    |
| 승인형 게시판 | 운영자 승인 후 활동 가능  |
| 비공개 게시판 | 초대받은 사용자만 접근 가능 |
| 공식 게시판  | 운영자가 인증한 게시판    |
| 홍보 게시판  | 추천 영역에 노출되는 게시판 |

## 9.4 게시판 DB 스키마

```sql
CREATE TABLE IF NOT EXISTS boards (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    owner_id INTEGER NOT NULL,
    name TEXT NOT NULL,
    slug TEXT NOT NULL UNIQUE,
    description TEXT,
    rules TEXT,
    visibility TEXT DEFAULT 'public',
    write_permission TEXT DEFAULT 'public',
    comment_permission TEXT DEFAULT 'public',
    cover_image TEXT,
    post_count INTEGER DEFAULT 0,
    member_count INTEGER DEFAULT 0,
    follower_count INTEGER DEFAULT 0,
    score INTEGER DEFAULT 0,
    is_official INTEGER DEFAULT 0,
    is_promoted INTEGER DEFAULT 0,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT,
    FOREIGN KEY (owner_id) REFERENCES users(id) ON DELETE CASCADE
);
```

## 9.5 게시판 멤버 테이블

```sql
CREATE TABLE IF NOT EXISTS board_members (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    board_id INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    role TEXT DEFAULT 'member',
    status TEXT DEFAULT 'active',
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(board_id, user_id),
    FOREIGN KEY (board_id) REFERENCES boards(id) ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

## 11. 게시판 홍보 시스템

## 11.1 홍보 기능 목적

게시판 운영자가 자신의 게시판을 노출하고, 신규 사용자를 모을 수 있도록 합니다.

```text
게시판 운영자
  ↓
게시판 홍보 신청
  ↓
홍보 카드 생성
  ↓
탐색 페이지, 홈 피드, 추천 영역 노출
  ↓
신규 사용자 유입
```

---

## 11.2 홍보 방식

| 방식       | 설명                |
| -------- | ----------------- |
| 무료 홍보    | 신규 게시판을 일정 기간 노출  |
| 인기 기반 노출 | 활동량이 높은 게시판 자동 노출 |
| 태그 기반 노출 | 관심 태그 사용자에게 노출    |
| 상호 홍보    | 게시판끼리 서로 추천       |
| 운영자 추천   | 관리자가 직접 선정        |
| 공식 추천    | 플랫폼 메인에 노출        |

## 11.4 게시판 홍보 신청 DB

```sql
CREATE TABLE IF NOT EXISTS board_promotions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    board_id INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    title TEXT NOT NULL,

);
```

## 16. 운영 안정화

## 16.1 Gunicorn 운영 기준

Oracle 서버에서 Flask를 직접 실행하지 않고 Gunicorn으로 실행합니다.

예시:

```bash
gunicorn -w 2 -k gthread --threads 4 -b 127.0.0.1:8000 wsgi:app
```

### 초기 권장값

| 항목           | 값              |
| ------------ | -------------- |
| workers      | 2              |
| threads      | 4              |
| bind         | 127.0.0.1:8000 |
| timeout      | 60초            |
| max requests | 1000           |

## 28. 개발 기준

고도화 작업 시 다음 기준을 유지합니다.

1. 기존의 단순하고 빠른 SNS 감각을 유지한다.
2. 게시글 작성은 계속 제목 없이 본문 중심으로 유지한다.
3. 모바일에서 가로 스크롤이 생기지 않도록 한다.
4. 게시판 탐색은 로그인하지 않아도 가능하게 한다.
5. 글 작성, 댓글, 좋아요, 팔로우는 로그인 후 가능하게 한다.
6. 게시판 홍보는 무분별하게 노출하지 않고 점수와 승인 기준을 둔다.
7. 운영 DB는 GitHub에 올리지 않는다.
8. 실데이터 운영은 Vercel이 아니라 Oracle 서버에서 한다.
9. 사용자가 늘어날수록 SQLite에서 PostgreSQL 전환을 우선 검토한다.
10. 관리자 기능과 신고 기능은 유저 증가 전에 먼저 준비한다.

## 30. 최종 정리

현재 SNS 게시판은 기본 기능이 완성된 MVP입니다.
이번 고도화의 목표는 이 MVP를 실제 유저가 모이고 활동할 수 있는 커뮤니티 플랫폼으로 확장하는 것입니다.

단순히 글쓰기와 댓글 기능만 있는 게시판은 사용자가 직접 찾아오기 어렵습니다.
따라서 게시판을 만들고, 홍보하고, 서로 추천하고, 사용자가 관심 게시판을 쉽게 찾는 구조가 필요합니다.

1차로는 게시판 시스템, 게시판 탐색, 게시판 홍보 기능을 추가합니다.
2차로는 PostgreSQL 전환, Nginx, Gunicorn, 백업, 로그, 관리자 기능을 통해 운영 안정성을 확보합니다.
3차로는 추천 알고리즘, 인기 게시판, 급상승 게시판, 태그 기반 탐색을 통해 유저 유입과 체류 시간을 늘립니다.

이 구조까지 완성되면 단순 SNS 게시판이 아니라, 여러 소규모 커뮤니티가 모이는 플랫폼 형태로 성장할 수 있습니다.
