# Telegram Style SNS Full Advanced Build Command MD

## 0. 작업 명령

기존 Flask 기반 Telegram Style SNS 프로젝트를 전체 고도화해줘.

목표는 텔레그램처럼 간단하고 깔끔한 내부용 SNS, 메신저 웹앱을 완성하는 것이다.

아래 요구사항을 기준으로 바로 실행 가능한 완성형 코드를 만들어줘.

중요 원칙:

* 기존 코드가 있으면 유지하면서 확장한다.
* 기존 DB가 있어도 깨지지 않게 migration 처리를 한다.
* 없는 파일은 새로 만든다.
* 필요한 폴더는 자동 생성한다.
* 코드 실행 후 바로 사용할 수 있어야 한다.
* 설명만 하지 말고 실제 코드를 수정하거나 생성한다.
* 사용자가 추가 질문 없이 바로 실행할 수 있도록 완성한다.
* 너무 복잡한 외부 서비스는 쓰지 않는다.
* 기본은 Flask, SQLite, HTML, CSS, JavaScript, Flask-SocketIO로 구현한다.

# 4. DB 설계

## 4.1 users 테이블

```text
users
- id INTEGER PRIMARY KEY AUTOINCREMENT
- username TEXT NOT NULL
- user_id TEXT NOT NULL UNIQUE
- password_hash TEXT NOT NULL
- profile_image TEXT
- status_message TEXT
- is_admin INTEGER DEFAULT 0
- is_active INTEGER DEFAULT 1
- created_at TEXT NOT NULL
- updated_at TEXT
```

규칙:

* user_id는 로그인용 아이디다.
* username은 화면 표시용 이름이다.
* password_hash는 반드시 해시로 저장한다.
* is_admin이 1이면 관리자다.
* is_active가 0이면 로그인 불가다.

---

## 4.2 rooms 테이블

```text
rooms
- id INTEGER PRIMARY KEY AUTOINCREMENT
- name TEXT NOT NULL
- description TEXT
- room_type TEXT DEFAULT 'group'
- created_by INTEGER
- is_notice INTEGER DEFAULT 0
- created_at TEXT NOT NULL
- updated_at TEXT
```

room_type 값:

```text
group
direct
notice
```

규칙:

* General 방은 group 타입이다.
* 1:1 채팅은 direct 타입이다.
* 공지방은 notice 타입이다.
* General 방은 DB 초기화 시 자동 생성한다.

---

## 4.3 room_members 테이블

```text
room_members
- id INTEGER PRIMARY KEY AUTOINCREMENT
- room_id INTEGER NOT NULL
- user_id INTEGER NOT NULL
- role TEXT DEFAULT 'member'
- is_pinned INTEGER DEFAULT 0
- is_favorite INTEGER DEFAULT 0
- last_read_message_id INTEGER DEFAULT 0
- joined_at TEXT NOT NULL
```

role 값:

```text
owner
admin
member
```

규칙:

* 방 생성자는 owner다.
* 일반 초대자는 member다.
* 채팅방별 읽음 위치는 last_read_message_id로 관리한다.
* 고정 여부는 is_pinned로 관리한다.
* 즐겨찾기는 is_favorite으로 관리한다.

---

## 4.4 direct_rooms 테이블

```text
direct_rooms
- id INTEGER PRIMARY KEY AUTOINCREMENT
- room_id INTEGER NOT NULL
- user1_id INTEGER NOT NULL
- user2_id INTEGER NOT NULL
- created_at TEXT NOT NULL
```

규칙:

* 1:1 채팅 중복 생성을 막기 위한 테이블이다.
* user1_id, user2_id는 작은 id를 user1_id에 저장한다.
* 같은 두 사용자 사이에는 하나의 direct room만 존재해야 한다.

---

## 4.5 messages 테이블

```text
messages
- id INTEGER PRIMARY KEY AUTOINCREMENT
- room_id INTEGER NOT NULL
- user_id INTEGER NOT NULL
- content TEXT
- attachment_id INTEGER
- reply_to_id INTEGER
- is_edited INTEGER DEFAULT 0
- is_deleted INTEGER DEFAULT 0
- is_pinned INTEGER DEFAULT 0
- created_at TEXT NOT NULL
- updated_at TEXT
```

규칙:

* content와 attachment_id가 모두 없으면 저장하지 않는다.
* 삭제 메시지는 실제 DB 삭제 대신 is_deleted를 1로 변경한다.
* 삭제된 메시지는 "Deleted message"로 표시한다.
* 수정된 메시지는 edited 표시를 붙인다.
* reply_to_id가 있으면 답장 메시지로 표시한다.

---

## 4.6 attachments 테이블

```text
attachments
- id INTEGER PRIMARY KEY AUTOINCREMENT
- original_name TEXT NOT NULL
- saved_name TEXT NOT NULL
- file_path TEXT NOT NULL
- file_url TEXT NOT NULL
- file_type TEXT
- file_size INTEGER
- uploaded_by INTEGER NOT NULL
- created_at TEXT NOT NULL
```

규칙:

* original_name은 사용자가 올린 원래 파일명이다.
* saved_name은 서버 저장 파일명이다.
* file_url은 브라우저에서 접근할 URL이다.
* 파일명은 secure_filename과 uuid를 사용해 안전하게 저장한다.

---

## 4.7 notifications 테이블

```text
notifications
- id INTEGER PRIMARY KEY AUTOINCREMENT
- user_id INTEGER NOT NULL
- room_id INTEGER
- message_id INTEGER
- title TEXT
- body TEXT
- is_read INTEGER DEFAULT 0
- created_at TEXT NOT NULL
```

규칙:

* 새 메시지 알림을 저장한다.
* 현재 접속 중이 아니어도 나중에 알림 목록을 볼 수 있게 한다.
* 이번 버전에서는 브라우저 푸시 알림은 제외하고 앱 내부 알림만 구현한다.

# 5. Migration 기준

기존 DB가 있어도 오류가 나지 않도록 `init_db()`에서 migration을 처리해줘.

필수 조건:

* 테이블이 없으면 생성한다.
* 컬럼이 없으면 ALTER TABLE로 추가한다.
* 이미 존재하는 컬럼이면 무시한다.
* General 방이 없으면 자동 생성한다.
* 기존 messages 테이블에 attachment_id, reply_to_id, is_edited, is_deleted, is_pinned, updated_at 컬럼이 없으면 추가한다.
* 기존 users 테이블에 profile_image, status_message, is_admin, is_active, updated_at 컬럼이 없으면 추가한다.
* 기존 rooms 테이블에 description, room_type, is_notice, updated_at 컬럼이 없으면 추가한다.

# 7. Route 설계

## 7.1 Page Routes

```text
GET /
GET /login
GET /register
GET /chat
GET /profile
GET /admin
GET /logout
```

규칙:

* `/`는 로그인 상태면 `/chat`, 아니면 `/login`으로 이동한다.
* `/chat`은 로그인 사용자만 접근 가능하다.
* `/profile`은 로그인 사용자만 접근 가능하다.
* `/admin`은 관리자만 접근 가능하다.

---

## 7.2 Auth Routes

```text
POST /login
POST /register
GET /logout
```

로그인 성공 응답:

```json
{
  "ok": true,
  "redirect": "/chat"
}
```

로그인 실패 응답:

```json
{
  "ok": false,
  "error": "Invalid user ID or password."
}
```

회원가입 성공 후 로그인 페이지로 이동한다.

---

## 7.3 User API

```text
GET  /api/me
GET  /api/users/search?q=keyword
GET  /api/users/<user_id>
```

### GET /api/me

현재 로그인 사용자 정보를 반환한다.

응답 예시:

```json
{
  "ok": true,
  "user": {
    "id": 1,
    "username": "Insup",
    "user_id": "insup",
    "profile_image": "/uploads/profiles/profile.png",
    "status_message": "Available",
    "is_admin": 1
  }
}
```

### GET /api/users/search

사용자 검색 기능이다.

규칙:

* username과 user_id에서 검색한다.
* 본인도 검색 가능하다.
* 비활성 사용자도 관리자 외에는 표시하지 않는다.

---

## 7.4 Room API

```text
GET    /api/rooms
POST   /api/rooms
GET    /api/rooms/search?q=keyword
GET    /api/rooms/<room_id>
PUT    /api/rooms/<room_id>
DELETE /api/rooms/<room_id>
POST   /api/rooms/<room_id>/members
GET    /api/rooms/<room_id>/members
DELETE /api/rooms/<room_id>/members/<user_id>
POST   /api/rooms/<room_id>/pin
POST   /api/rooms/<room_id>/favorite
POST   /api/rooms/<room_id>/read
POST   /api/direct/start
```

### GET /api/rooms

현재 사용자가 참여한 채팅방 목록을 반환한다.

각 채팅방에는 다음 정보를 포함한다.

```json
{
  "id": 1,
  "name": "General",
  "description": "Default room",
  "room_type": "group",
  "last_message": "Hello",
  "last_time": "10:30",
  "unread_count": 3,
  "is_pinned": 1,
  "is_favorite": 0
}
```

정렬 기준:

1. 고정 채팅방 먼저
2. 마지막 메시지 시간이 최신인 채팅방 먼저
3. 메시지가 없으면 생성일 최신순

### POST /api/rooms

그룹 채팅방 생성 API다.

요청:

```json
{
  "name": "Team",
  "description": "Team chat"
}
```

생성자는 자동으로 owner로 room_members에 추가한다.

### GET /api/rooms/search

현재 사용자가 참여한 채팅방 중 이름으로 검색한다.

규칙:

* 대소문자 구분 없이 검색
* 검색어가 비면 전체 목록 반환
* debounce는 frontend에서 300ms 적용

### POST /api/direct/start

1:1 채팅 시작 API다.

요청:

```json
{
  "target_user_id": 2
}
```

응답:

```json
{
  "ok": true,
  "room": {
    "id": 5,
    "name": "David",
    "room_type": "direct"
  }
}
```

규칙:

* 같은 사용자와 기존 1:1 방이 있으면 기존 방을 반환한다.
* 없으면 새 direct room을 생성한다.
* direct_rooms 테이블에 중복되지 않게 저장한다.

* 현재 사용자가 참여한 방에서만 검색 가능
* content 기준 검색
* 삭제 메시지는 검색 제외
* 검색어가 비면 최근 메시지 반환

### PUT /api/messages/<message_id>

메시지 수정 API다.

규칙:

* 본인 메시지만 수정 가능
* 삭제된 메시지는 수정 불가
* 첨부파일은 수정하지 않고 텍스트만 수정
* 수정 후 is_edited를 1로 설정

### DELETE /api/messages/<message_id>

메시지 삭제 API다.

규칙:

* 본인 메시지 또는 관리자만 삭제 가능
* 실제 삭제하지 않고 is_deleted를 1로 변경
* content는 빈 값 또는 Deleted message로 처리
* attachment_id는 null 처리해도 된다

### POST /api/messages/<message_id>/pin

메시지 고정 또는 고정 해제 API다.

규칙:

* 방 owner 또는 admin만 가능
* is_pinned 값을 변경한다

규칙:

# 21. 코드 작성 규칙

* app.py는 주석으로 구역을 나눠줘.
* storage.py에는 DB 함수만 작성해줘.
* chat.js는 기능별 함수로 나눠줘.
* CSS는 static/css/style.css에만 정리해줘.
* HTML에는 필요한 data attribute를 적절히 넣어줘.
* 중복 코드는 함수로 정리해줘.
* 에러는 화면에 표시해줘.
* console.log는 개발에 필요한 최소한만 남겨줘.
* Python 예외 발생 시 서버가 죽지 않게 처리해줘.
* 사용자가 없는 값, 빈 값, 잘못된 값을 보내도 안전하게 처리해줘.
